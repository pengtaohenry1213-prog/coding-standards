# Human Gate 完整可行方案

## 方案背景

当前项目 Human Gate 定义散落于多个文档，存在以下不足：

- AGENTS.md 定义了门禁输入项，但未细化触发时机和执行流程
- Owner 角色未明确可用性承诺
- CI 集成仍为 TODO 项
- 缺乏分阶段实施路径

---

## 一、门禁阶段设计

### 1.1 五级门禁架构

```
┌─────────────────────────────────────────────────────────────┐
│  Gate 0: 需求冻结门（Pre-Coding）                           │
│  Owner 决策：需求是否可以进入契约设计？                       │
├─────────────────────────────────────────────────────────────┤
│  Gate 1: 契约冻结门（API Design Freeze）                    │
│  Owner 决策：API 契约是否可以进入编码？                      │
├─────────────────────────────────────────────────────────────┤
│  Gate 2: 编码完成门（Implementation Complete）              │
│  Owner 决策：实现质量是否可以进入测试？                      │
├─────────────────────────────────────────────────────────────┤
│  Gate 3: 测试完成门（Testing Complete）                     │
│  Owner 决策：测试结果是否可以进入发布？                      │
├─────────────────────────────────────────────────────────────┤
│  Gate 4: 发布放行门（Go/No-Go）                             │
│  Owner 决策：是否可以正式发布？                              │
└─────────────────────────────────────────────────────────────┘
```

## 二、Owner 可用性模型

### 2.1 异步决策模式（核心设计）

为解决 Owner 无法实时在线的问题，采用"异步 Gate"模式：

```
1. AI/Developer 提交 Gate 请求（PR/Issue）
2. Owner 在 SLA 周期内异步审批
3. 自动通知（邮件/IM/Slack）
4. 超时自动升级或默认推进
```

### 2.2 SLA 定义


| Gate 类型        | 正常响应 SLA | 紧急响应 SLA | 超时动作          |
| -------------- | -------- | -------- | ------------- |
| Gate 0/1 (冻结门) | 48 小时    | 24 小时    | 自动延期 24h      |
| Gate 2 (编码完成)  | 24 小时    | 12 小时    | 自动延期 12h      |
| Gate 3 (测试完成)  | 12 小时    | 4 小时     | 自动升级至备选 Owner |
| Gate 4 (发布放行)  | 4 小时     | 1 小时     | 必须等待，禁止自动推进   |


### 2.3 决策输入模板

Owner 在每个 Gate 需要做出决策：

```
## Gate [N] 决策单

### 基本信息
- 任务/需求ID：
- Gate 类型：
- 提交时间：
- Owner 接收时间：

### 检查结果
| 检查项 | 状态 | 备注 |
|--------|------|------|
| [从 Gate 检查清单复制] | Pass/Fail | |

### 决策选项
- [ ] Go（全部通过，可进入下一阶段）
- [ ] Conditional Go（部分通过，需满足条件清单）
- [ ] No-Go（有阻断项，返回修复）

### 决策理由
（必填，至少一句话）

### 签名
Owner: ____________  时间: ____________
```

### 2.4 备选 Owner 机制

- 每个 Owner 配置 1 名备选 Owner
- 紧急场景自动通知备选 Owner
- 备选 Owner 权限与主 Owner 相同
- 连续 2 次超时触发 Owner 评估

## 三、CI/CD 集成计划

### 3.1 分阶段 CI 接入

```
阶段 1（0-2 周）：仅提示，不阻断
├── 架构边界检查 → 输出警告日志
├── 可观测验证检查 → 输出建议
└── 静态检查 → 输出报告

阶段 2（3-4 周）：部分阻断
├── 架构边界检查 → PR 阻塞（可选）
├── 构建通过 → 必须通过
├── 契约测试 → 必须通过
└── P0/P1 缺陷 → 必须清零

阶段 3（5 周+）：完全阻断
├── 全部检查 → 必须通过
├── 性能基线 → TP99 必须达标
└── 安全扫描 → 无 P0/P1 漏洞
```

### 3.2 CI 检查矩阵


| 检查项      | 阶段1 | 阶段2  | 阶段3 |
| -------- | --- | ---- | --- |
| 构建       | 阻断  | 阻断   | 阻断  |
| 静态检查     | 提示  | 阻断   | 阻断  |
| 单元测试     | 提示  | 阻断   | 阻断  |
| 契约测试     | 提示  | 阻断   | 阻断  |
| 回归测试     | 提示  | 提示   | 阻断  |
| P0/P1 缺陷 | 提示  | 阻断   | 阻断  |
| 架构边界     | 提示  | 可选阻断 | 阻断  |
| 可观测验证    | 提示  | 提示   | 阻断  |
| 性能基线     | 不检查 | 不检查  | 阻断  |
| 安全扫描     | 提示  | 提示   | 阻断  |


### 3.3 CI 输出规范

每个检查输出统一格式：

```
[GATE-CHECK]
type: [架构边界/契约测试/可观测验证/...]
status: [PASS/FAIL/WARNING]
evidence: [链接或日志路径]
blocking: [true/false]
recommendation: [建议]
timestamp: [ISO时间]
```

## 四、决策流与升级机制

### 4.1 正常决策流

```
开发者提交 Gate 请求
    ↓
Coordinator 预审（1小时内）
    ↓
通知 Owner（自动）
    ↓
[等待 SLA]
    ├── Owner 决策 → 输出决策单
    ├── 超时 → 延期通知 → 继续等待
    └── 紧急升级 → 触发备选 Owner
    ↓
决策执行（自动）
```

### 4.2 升级触发条件


| 触发条件               | 升级动作           |
| ------------------ | -------------- |
| Owner SLA 超时 1.5 倍 | 通知主 + 备选 Owner |
| Owner 连续 2 次超时     | 记录到协调日志，评估替代方案 |
| 决策为 No-Go 且争议      | 召开紧急评审会议       |
| P0/P1 缺陷无法在当前迭代清零  | 升级至项目级别决策      |


### 4.3 争议解决

当 Developer 对 No-Go 决策有异议时：

1. Developer 在 24h 内提交书面申诉
2. Coordinator 协调二次评审
3. Owner 最终裁决（不可申诉）

## 六、度量与持续改进

### 6.1 关键度量指标


| 指标                | 定义                     | 目标          |
| ----------------- | ---------------------- | ----------- |
| Gate 响应时长         | 提交到决策完成的时间             | < SLA 的 80% |
| Gate 通过率          | Go / Total             | > 80%       |
| Conditional Go 比例 | Conditional Go / Total | < 15%       |
| No-Go 转 Go 率      | 修复后通过 / No-Go 总数       | > 90%       |
| CI 检查误报率          | 假阳性 / 总警告数             | < 10%       |
| 文档同步率             | 更新及时的 Gate 比例          | 100%        |


### 6.2 评审节奏


| 评审类型       | 频率       | 参与者                 |
| ---------- | -------- | ------------------- |
| 周度 Gate 复盘 | 每周       | Coordinator + Owner |
| 迭代 Gate 评审 | 每 Sprint | 全团队                 |
| 季度流程评估     | 每季度      | Owner + 项目管理层       |

## 九、总结

本方案提供：

1. **五级门禁架构**：Gate 0-4 对应需求→契约→编码→测试→发布全流程
2. **异步决策模式**：通过 SLA 和通知机制解决 Owner 可用性问题
3. **分阶段 CI**：渐进式接入，先提示后阻断
4. **完整实施路径**：3 阶段 8 周落地
5. **度量与改进机制**：持续优化 Gate 效率
