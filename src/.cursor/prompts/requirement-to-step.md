# 需求 → Step 转换流程

本 prompt 负责将业务需求文档转换为标准化的 stepN.md 文件。遵循 CO-STAR 框架的 Mode A 格式。

---

## 1. Context（背景）

需求到 Step 的转换是 AI 工程化开发的第一步。通过将业务需求拆解为标准化的 stepN.md，确保每个 step 可执行、可验收、可追溯。

---

## 2. Objective（目标）

将业务需求文档（PRD、需求描述、会议纪要等）转换为标准化的 stepN.md 文件。

---

## 3. Input（输入）

| 输入 | 说明 |
|------|------|
| 业务需求文档 | PRD、需求描述、会议纪要等 |

---

## 4. Process（转换流程）

### Step 1：需求分析

读取需求文档，提取以下关键信息：

| 信息类型 | 说明 | 示例 |
| --- | --- | --- |
| 背景 | 为什么要做这件事 | 用户需要批量导入数据 |
| 目标 | 要达成什么结果 | 支持 Excel 模板批量导入 |
| 用户场景 | 谁在什么情况下使用 | 财务人员在月初批量导入数据 |
| 功能范围 | 要做哪些功能 | Excel 解析、字段映射、数据校验 |
| 不做清单 | 明确不做的内容 | 不支持 CSV、不支持公式解析 |

---

### Step 2：Step 拆分

根据 `steps-dev.md` 的阶段划分原则，将需求拆分为多个 step：

| 阶段 | 适用场景 | Step 类型 |
| --- | --- | --- |
| 第零阶段 | 项目初始化、工程化配置 | step0 |
| 第一阶段 | 前后端基础项目初始化 | step1, step2 |
| 第二阶段 | 核心功能链路 | step3, step4, step5 |
| 第三阶段 | 集成与优化 | step6, step7 |

**拆分原则**：

- 一个 step = 一个独立可验收的单元
- step 之间尽量无依赖（或仅 minor 依赖）
- 每个 step 的工作量为 0.5-2 人天

---

### Step 3：生成 stepN.md

对照 `step-template.md` 模板，为每个 step 生成标准化的 step 文件。

---

### Step 4：自检清单

生成 stepN.md 后，对照以下清单自检：

| 检查项 | 说明 | 必须 |
| --- | --- | --- |
| 目标明确 | 用一句话能说清楚 step 要完成什么 | ✅ |
| 输入条件完整 | 前置资源/数据/环境都已明确 | ✅ |
| 执行动作具体 | 每个动作都是可执行的 | ✅ |
| 验收标准可测 | 有明确的验证方式和量化指标 | ✅ |
| 不做清单明确 | 明确标注了 Out of Scope | ✅ |
| 依赖关系清晰 | 前置/后续 step 都已标注 | ✅ |
| 测试用例覆盖 | 核心路径有对应的测试用例 | ✅ |

---

### Step 5：Human Gate PM 评审

将生成的 stepN.md 提交 Human Gate PM 评审：

| 检查项 | 说明 |
| --- | --- |
| 需求完整性 | 背景与目标清晰 |
| 验收标准 | 功能验收标准可执行、可观测 |
| 优先级 | P0 主路径清晰 |
| 依赖与风险 | 外部依赖已识别 |

**决策**：

| 决策 | 条件 |
| --- | --- |
| PASS | 通过评审，进入执行阶段 |
| REJECT | 修正后重新评审 |

---

## 5. Output（输出物）

| 输出 | 说明 |
|------|------|
| stepN.md | 每个拆分的 step 一个文件，输出到 `doc/steps/` |
| 阶段划分表更新 | 更新 `doc/steps-dev.md` 中的阶段划分表 |

---

## 6. Log Format（日志格式）

```
### HH:MM - [agent: planner]

- 操作：需求转 Step
- 输入：需求文档
- 执行：需求分析 → Step 拆分 → 生成 stepN.md → 自检 → PM 评审
- 结果：完成
```

---

## 7. Git Commit（Git 提交规范）

```
feat(steps): 生成 step{N} ~ step{M}
```

---

## 8. Hard Rules（强制规则）

- **stepN.md 必须通过 Human Gate PM 评审后才能执行**
- **不允许合并多个需求到一个 step**
- **验收标准必须量化**（不可量化的标准需拆分）
- **每个 step 必须有对应的测试用例**
