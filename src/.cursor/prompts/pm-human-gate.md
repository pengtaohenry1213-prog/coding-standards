# Human Gate：PM 需求评审

本 prompt 负责对 Plan/PRD 进行需求评审。遵循 CO-STAR 框架的 Mode B 格式。

---

## 1. Context（背景）

PM Human Gate 是 Human Gate 双审机制的第一道关卡，在 Step 执行前对需求和 Plan 进行评审，确保需求完整、验收标准明确、优先级合理。

### 触发时机

| 时机 | 说明 |
|------|------|
| Plan/PRD 提交评审时 | Plan 生成后、执行前 |
| 需求变更范围≥20%时 | 范围变更需重新评审 |
| 里程碑节点验收时 | 阶段验收时的需求确认 |

---

## 2. Objective（审核目标）

确保需求完整、验收标准可执行、优先级合理、依赖与风险已识别。

---

## 3. Checklist（审核清单）

### 1. 需求完整性

- [ ] 背景与目标清晰（可量化更好）
- [ ] **不做清单**明确（Out of Scope）
- [ ] 用户与场景定义完整

### 2. 验收标准

- [ ] 功能验收标准可执行、可观测
- [ ] 非功能需求明确（性能/安全/兼容）
- [ ] 异常与边界场景覆盖

### 3. 优先级

- [ ] P0 缺失则核心价值不成立
- [ ] P1 主路径清晰
- [ ] P2/P3 有则说明

### 4. 依赖与风险

- [ ] 外部依赖（数据/接口/法务）已识别
- [ ] 开放问题有责任人
- [ ] 风险有应对方案

---

## 4. Decision（决策格式）

| 决策 | 条件 |
|------|------|
| **PASS** | 所有检查项通过，可继续执行 |
| **CONDITIONAL** | 非核心项未达标，但不影响主路径，需记录整改项 |
| **REJECT** | 核心项缺失或不明确，必须修复后才能继续 |

---

```markdown

|------|------|

| 决策依据 | ... |
| 整改要求（如有） | ... |
| 整改期限（如有） | ... |
| 签字 | ___ |
```

---

## 6. Closure（闭环要求）

- 决策需在提交后 **24h 内**给出
- CONDITIONAL/REJECT 需跟踪至整改完成并重新决策

---

## 7. Prohibitions（禁止事项）

- 无验收标准的 Plan 不得通过
- 无 Out of Scope 定义的 Plan 不得通过
- P0 缺失的 Plan 不得通过
